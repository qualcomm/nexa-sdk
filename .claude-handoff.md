# Handoff: cross-plugin HTP session lifecycle fix

Written 2026-05-13 on Windows-on-Snapdragon host (arm64) for continuation on a linux/amd64 workstation.

## What the bug is

If you load a llama.cpp GGUF model on NPU (`plugin_id=llama_cpp`, `device_id=npu` or `hybrid`), unload it, then load a QAIRT model — the first QAIRT load succeeds. But unloading the QAIRT model and re-loading it fails with QNN `err 1007` (`CONTEXT_ERROR_CREATE_FROM_BINARY_FAILED`) at `third-party/geniex-qairt/qnn-api/src/QnnApi.cpp:1469`. Symmetric failure mode exists for the reverse order.

Full reproduction matrix is in the original conversation, but the common factor is: any time both plugins have touched the HTP in the same process, QAIRT's second re-init fails.

## Root cause

`third-party/llama.cpp/ggml/src/ggml-hexagon/ggml-hexagon.cpp:3359-3382` — `ggml_backend_hexagon_reg()` is a process-wide singleton. It constructs `ggml_hexagon_session`s holding open FastRPC channels to the CDSP (visible in logcat as handles to `libggml-htp-v79.so`). Nothing ever tears these down; `ggml_backend_hexagon_free` (line 2604) only deletes the per-backend wrapper.

Meanwhile `third-party/geniex-qairt/qnn-api/src/QnnApi.cpp:87-91` deliberately skips `dlclose` on the HTP backend library in `~QnnApi`. QAIRT tolerates a stale HTP state on the first `initializeHtp` (cold-ish) but the second `initializeHtp` after QAIRT's own teardown trips the context-create-from-binary failure because the DSP already has state registered from llama.cpp's still-alive session.

## Fix approach

Let the llama.cpp plugin release its HTP sessions (closing FastRPC channels) when its last LlamaLlm is destroyed. Sessions are lazily re-created by the HTP registry on the next llama.cpp load, so it's transparent to callers.

## What's already in place on the Windows host (also works on linux — files are portable)

Three files in `sdk/` are modified:

1. **`sdk/patches/llama-hexagon-release-sessions.patch`** (new) — patches llama.cpp to:
   - add `ggml_backend_hexagon_release_sessions()` (destroys all `ggml_hexagon_session`s but leaves the registry/`devices[]` array intact)
   - make `ggml_backend_hexagon_reg_get_device()` lazily recreate a session when its context is null
   - expose the release function via `get_proc_address` under the name `"ggml_backend_hexagon_release_sessions"`

2. **`sdk/CMakeLists.txt`** — adds a configure-time patch-apply step under `GENIEX_PLUGIN_LLAMA_CPP`. Uses `git apply --reverse --check` first to detect an already-patched submodule so re-configuration is a no-op. Marker file is `<build>/.llama-hexagon-release-sessions.applied`.

3. **`sdk/plugins/llama_cpp/{include/llm.h, src/llm.cpp}`** — adds:
   - `bool uses_htp` on `LlamaLlm` (llm.h)
   - a static `std::atomic<int> g_htp_refcount` in llm.cpp (anonymous namespace)
   - in `create_impl`, detects HTP among the selected devices by asking each `ggml_backend_dev_t` for its backend reg name (`"HTP"`), or assumes HTP when `mpar.n_gpu_layers != 0` (hybrid path). Bumps the refcount.
   - in `~LlamaLlm`, if `uses_htp` was set, decrements and — on last release — calls the new `ggml_backend_hexagon_release_sessions` via `ggml_backend_reg_get_proc_address(ggml_backend_reg_by_name("HTP"), ...)`.

## Current state of submodule

`third-party/llama.cpp` is at upstream commit `227ed28e1` with the patch **applied in-tree but not committed**. If you run the build from a fresh clone on the linux workstation, the CMake configure step will apply the patch automatically (it invokes `git apply` against the submodule). Nothing else to do.

The applied diff has been dumped to `.claude-handoff-diff.patch` in the repo root as a reference.

## How to build on the linux workstation

From the repo root, just run the already-documented Android build exactly as in [notes/build.md § Android (cross-compile from Linux)](notes/build.md):

```bash
docker run --rm -u $(id -u):$(id -g) \
    --volume $(pwd):/workspace \
    --workdir /workspace/sdk \
    -e CCACHE_DIR=/workspace/.ccache \
    --platform linux/amd64 \
    ghcr.io/qcom-ai-hub/geniex-toolchain-android:v0.0.1 \
    bash -c 'cmake --preset arm64-android-snapdragon-debug -B build-android . \
      && cmake --build build-android -j \
      && cmake --install build-android --prefix pkg-geniex'
```

This produces `sdk/pkg-geniex/lib/arm64-v8a/` with the patched `libggml-hexagon.so` and `libgeniex_plugin_llama_cpp.so`. No change to the wrapper steps.

Then build the demo APK from `examples/android/`:

```bash
cd examples/android
./gradlew assembleDebug
```

(Gradle will pick up the updated native libs via the existing `copyBridgeLibs` task in `app/build.gradle.kts`.)

## What to verify on-device after build

Install APK, run the 7-case matrix from the original bug report, especially these two that used to fail:

- Case 6: qairt → unload → llama.cpp(npu) → unload → qairt → unload → **qairt** (reload must succeed now)
- Case 7: llama.cpp(npu) → unload → qairt → unload → **qairt** (reload must succeed now)

In logcat, look for this new line on each llama.cpp unload when it's the last HTP user:
```
D/GeniexSdk: Releasing HTP sessions on last llama.cpp HTP handle destroy
```
followed by ggml-hex's own release log:
```
I/ggml-hex: releasing session: HTP0
```

Failure signature (what we're fixing — should NOT appear on re-load):
```
E/jni: [STDERR] [ERROR] "Could not create context from binary for context index = 0 : err 1007"
E/jni: [STDERR] [ERROR] "Create From Binary FAILED!"
E/GeniexSdk: Failed to create QAIRT LLM pipeline for model: ...
```

## Risks still open

- `ggml_hexagon_session::release()` has `GGML_ABORT` calls on `htp_iface_stop` or `dspqueue_close` failure (`third-party/llama.cpp/ggml/src/ggml-hexagon/ggml-hexagon.cpp:2052, 2066`). These are latent on process-exit today; our change makes them fire mid-session during an unload. If abort fires, investigate why `htp_iface_stop`/`dspqueue_close` are failing — likely a model still in-flight on the DSP, which means the decoder thread hasn't joined before the destructor runs.
- Patch is against llama.cpp `227ed28e1`. Future submodule bumps need the patch re-generated against the new base. CMake prints `FATAL_ERROR` with instructions if `git apply` fails.

## What the toolchain image is, and why the switch away from llama.cpp's Snapdragon image

**`ghcr.io/qcom-ai-hub/geniex-toolchain-android:v0.0.1`** is defined in `.github/docker/toolchain-android.Dockerfile`. It's a thin layer on top of upstream `ghcr.io/snapdragon-toolchain/arm64-android:v0.3` (the same image you used previously for llama.cpp Hexagon builds). Contents:

Base (`snapdragon-toolchain/arm64-android:v0.3`):
- Android NDK r29
- Hexagon SDK (required for signing HTP skels: `libggml-htp-v68.so` through `v81.so`)
- OpenCL headers/libs for GGML's OpenCL backend
- CMake, Ninja

Our derived layer adds:
- `build-essential` — for Rust proc-macro **host** linking during cross-compile (cargo builds proc-macros for the host arch even when `--target=aarch64-linux-android`; without a host C compiler, the build errors out before it gets to any Android code)
- `ccache` — we redirect `CCACHE_DIR` to `$repo/.ccache` for fast re-builds
- `rustup` with the stable toolchain + `rustup target add aarch64-linux-android` — baked in so per-run `rustup target add` is a no-op. Needed by the Rust model-manager crate at `sdk/model-manager/`, which is statically linked into `libgeniex.so`.

**Why the build command is different vs. your llama.cpp-only build**:
- llama.cpp on its own needs only NDK + Hexagon SDK + CMake — the snapdragon base image alone covers that. You don't need Rust.
- GenieX additionally bundles a Rust-written model manager (HF/AI Hub downloader + local model registry). Without the Rust tooling pre-installed, every build needs to bootstrap rustup + target + proc-macro cross-compile, which is what took that 1h the first time.
- The custom layer exists specifically so CI and you don't pay that cost repeatedly. On a fresh workstation the first `docker pull` is ~8 GB (same as the base image) — subsequent builds reuse the image layer cache.

**On Windows-on-Snapdragon** (this host): the image is `linux/amd64`-only because the Hexagon SDK signing tools are x86-64 binaries. Docker Desktop for Windows-on-ARM only exposes `linux/arm64`, so the image fails with `exec format error` unless you install binfmt/QEMU — and emulated amd64 on ARM Snapdragon would be very slow. On a native linux/amd64 workstation, none of that applies.

## If you want the patch committed vs. applied-on-configure

Right now the SDK CMake does a runtime `git apply`. If you prefer the submodule commit to land the patch permanently (cleaner git log, but forks llama.cpp), steps are:

1. `cd third-party/llama.cpp && git checkout -b geniex/hexagon-release-sessions && git add -u && git commit -m "ggml-hex: expose release_sessions for cross-plugin HTP handoff"`
2. Push the branch to a qcom-ai-hub fork of llama.cpp.
3. Edit `.gitmodules` to repoint the submodule.
4. Update the submodule pointer in the parent repo.
5. Remove the CMake patch-apply block and `sdk/patches/llama-hexagon-release-sessions.patch`.

This was offered and you chose the build-time patch approach instead, so the repo is set up for that. If you change your mind, the patch is already in `.claude-handoff-diff.patch` form ready to commit.
