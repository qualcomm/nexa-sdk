Quantization choice depends on which runtime you're using.

## **llama.cpp**

llama.cpp accepts the full range of GGML quantization formats. The CLI prompts you to pick one when you `geniex pull` a GGUF model:

```powershell
Choose a precision version to download
> Q4_0       [1.2 GiB] (default)
  Q8_0       [2.0 GiB]
  F16        [3.8 GiB]
```

| Precision | Use when |
|---|---|
| **`Q4_0`** *(recommended on Snapdragon)* | Best Hexagon NPU support in llama.cpp. Default for most models. |
| `Q8_0` | Better quality at ~2× the disk and memory cost. Falls back to GPU/CPU. |
| `F16` | Reference precision. Mainly for evaluation — large and slow on NPU. |
| `Q4_K_M`, `Q5_K_M`, etc. | Mixed-precision K-quants. Run on GPU/CPU; not optimized for Hexagon NPU. |

<Tip>Stick with `Q4_0` if you want the model to land on the Hexagon NPU. Other precisions will work but typically run on GPU or CPU.</Tip>

## **QAIRT**

QAIRT models come **pre-quantized** in the AI Hub bundle — there is no quantization choice at runtime. The bundle is compiled for a specific Snapdragon chipset and target precision; the KV cache and context length are also baked in.

If you need a different precision for a QAIRT model, you need a different bundle from Qualcomm AI Hub.
