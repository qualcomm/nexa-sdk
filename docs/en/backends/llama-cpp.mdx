The `llama_cpp` backend runs **any GGUF model** through llama.cpp with Qualcomm's GGML Hexagon backend. It's the broadest path — you can pull any community GGUF from Hugging Face and run it on Snapdragon NPU, Adreno GPU, or pure CPU.

## **Devices**

| Device alias | Effect |
|---|---|
| `npu` | Pin to Hexagon NPU (`HTP0`). Best NPU-only path. |
| `gpu` | Adreno GPU via OpenCL. |
| `cpu` | Pure CPU. Forces `nGpuLayers = 0`. |
| `hybrid` *(default)* | Empty `device_id` + `n_gpu_layers=999` — llama.cpp's per-tensor HTP+CPU scheduler. **The fast path on Snapdragon.** |

If you don't pass `--device`, `llama_cpp` picks `hybrid`. Some model families override this — for example, anything matching `gpt-oss` falls back to `npu` because it can't run on the hybrid path.

## **Quantization**

llama.cpp supports the full GGML quantization range. **Use `Q4_0` for best Hexagon NPU support** — other quants run on GPU/CPU. See [Quantization](/en/models/quantization).

## **Picking a model**

Run any GGUF model with:

```powershell windows
geniex infer <org>/<repo>-GGUF
```

When prompted:
- **Model type** — `vlm` for Qwen3-VL family, `llm` otherwise.
- **Quantization** — `Q4_0` recommended.

See [Supported models](/en/models/supported#llamacpp-gguf) for tested examples and [Bring your own model](/en/models/bring-your-own) for arbitrary GGUFs.

## **Limitations**

- Multimodal support for `Qwen3.5` and `Gemma4` is not yet available — pick `llm` when prompted.
- `gpt-oss` family cannot run on the hybrid path; the runtime falls back to `npu` automatically.
- Quants outside `Q4_0` typically don't land on Hexagon NPU and will use GPU/CPU instead.
